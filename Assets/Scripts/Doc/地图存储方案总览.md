# 地图存储方案总览

## 1. 结论先行

当前地图系统并不是“纯 Tilemap 存档”，而是一个由 `WorldState` 统一管理的双层数据模型：

- 第一层是 `BasicMapData`，用一维数组保存“每个格子的基础状态”。
- 第二层是 `DetailedMapData`，用 `Dictionary<Vector3Int, DetailedCellData>` 只保存“少量需要额外状态的格子”。
- 同时再配合 `Entitys` 字典，把格子上的运行时实体映射到具体的 `IEntityRuntime` 实例。

也就是说，Tilemap 主要负责视觉显示，真正的逻辑状态仍然由 `WorldState` 内存结构承载。

## 2. 核心入口

地图相关的主入口在 `Assets/Scripts/Data/WorldState.cs`：

- `MapSize` 定义逻辑地图尺寸。
- `MainTile`、`OverlapTile`、`UperTile` 分别表示三层瓦片显示层。
- `BasicMapData`、`DetailedMapData`、`Entitys` 共同构成地图状态。

对应代码位置：

- `WorldState` 字段区：`WorldState.cs:74-86`
- 地图初始化：`WorldState.cs:122-141`
- 基础查询：`WorldState.cs:163-185`
- 实体放置与瓦片写入：`WorldState.cs:186-303`
- 详细格子数据管理：`WorldState.cs:330-379`
- 交互与生命周期：`WorldState.cs:418-510`

## 3. 数据结构拆解

### 3.1 `BasicCellData`：全图基础层

`BasicCellData` 是一个结构体，保存每个格子最基础的信息：

- `Type`：格子类型，当前枚举有 `Water / Soil / Farmland / Grass`
- `Empty`：该格子是否允许继续放置内容

对应代码：

- 枚举定义：`WorldState.cs:8-14`
- 结构定义：`WorldState.cs:15-31`

初始化规则：

- 默认类型是 `Soil`
- 如果类型是 `Water`，则 `Empty = false`
- 其他类型默认 `Empty = true`

这里体现出当前设计思路是：

- `Type` 负责描述“土地性质”
- `Empty` 负责描述“占用判定”

这两个字段是分离的，不是由同一个值推导出来。

### 3.2 `BasicMapData`：一维数组表示二维地图

`BasicMapData` 的定义如下：

```csharp
private BasicCellData[] BasicMapData;
```

初始化方式：

```csharp
BasicMapData = new BasicCellData[lenth * height];
```

二维坐标到一维索引的映射方式：

```csharp
index = y * MapSize.x + x;
```

对应代码：

- 初始化：`WorldState.cs:133-141`
- 索引换算：`WorldState.cs:171-175`

这说明当前地图底层不是按“块”或“分区”存储，而是一个完整平铺的大数组。

优点：

- 查询快
- 内存连续
- `CheckEmpty` 这类高频逻辑成本低

限制：

- 逻辑上默认地图是一个规则矩形
- 如果将来做超大地图、分区流式加载、无限地图，就需要重构

### 3.3 `DetailedCellData`：稀疏扩展层

`DetailedCellData` 用来保存“不是每个格子都需要”的附加信息：

- `EntityID`：当前格子挂着哪些实体 ID
- `WaterTimeLast`：含水剩余时间
- `FertTimeLast`：肥料剩余时间
- `FertRank`：肥料等级

对应代码：

- 定义：`WorldState.cs:33-64`
- 存储容器：`WorldState.cs:84-85`

这里的设计是典型的“稀疏存储”：

- 普通空地只进 `BasicMapData`
- 只有被实体、湿润、施肥等状态影响的格子，才进入 `DetailedMapData`

因此 `DetailedMapData` 更像是“地图状态补丁层”。

### 3.4 `Entitys`：实体运行时注册表

`WorldState` 还维护了：

```csharp
public Dictionary<int, IEntityRuntime> Entitys { get; private set; } = new();
```

对应代码：

- 定义：`WorldState.cs:86`
- 注册：`WorldState.cs:143-161`
- 查询：`WorldState.cs:320-327`

职责不是存“地图格子数据”，而是存“实体对象实例”：

- 格子只记 `EntityID`
- 真正的行为逻辑在 `IEntityRuntime` 对象里
- 通过 `GetEntity(entityID)` 再跳转到实体实例

这相当于把“空间索引”和“行为对象”分离了。

## 4. 三层 Tilemap 的角色

当前视觉层由三张 Tilemap 组成：

- `MainTile`
- `OverlapTile`
- `UperTile`

对应代码：`WorldState.cs:78-81`

目前已知调用习惯：

- `PlaceTile(..., tileLayer, ...)` 按层写入瓦片：`WorldState.cs:259-283`
- `SwitchTile(..., tileLayer)` 用于替换显示：`WorldState.cs:284-303`
- 锄地时农田底图写在第 `1` 层：`WorldState.cs:489`
- 作物生长瓦片写在第 `2` 层：`Farmland_Entity.cs:93-101`、`Crops_Entity.cs:76`

因此可以把三层理解为：

- `MainTile`：基础地表
- `OverlapTile`：覆盖层，例如农田地表
- `UperTile`：更上层的表现，例如作物阶段图

## 5. 地图初始化流程

`WorldState.Awake()` 会在场景启动时完成两件事：

1. `InitMap(MapSize.x, MapSize.y)`
2. `ItemRegistry.Init(ItemDatabase)`

对应代码：`WorldState.cs:121-131`

地图初始化实际只做了以下内容：

1. 分配 `BasicMapData`
2. 把所有格子初始化为默认 `BasicCellData.Create()`
3. 缓存 `MainTile.cellSize`

注意：

- 当前没有从 Tilemap 反向扫描地形生成 `BasicMapData`
- 也没有读取任何序列化存档
- 因此当前逻辑地图初始状态几乎完全依赖代码默认值，而不是场景瓦片内容

## 6. 查询与坐标转换

### 6.1 世界坐标与格子坐标

`WorldState` 直接复用了 `MainTile` 的换算能力：

- `WorldToCell(Vector3 worldPos)`：`WorldState.cs:163-166`
- `CellToWorld(Vector3Int cellPos)`：`WorldState.cs:167-170`

这意味着：

- 逻辑坐标系与 `MainTile` 的 Grid 保持一致
- 其他层必须和 `MainTile` 对齐，否则逻辑和显示会错位

### 6.2 占用判断

`CheckEmpty` 的流程非常直接：

1. 用 `Index(x, y)` 把坐标映射到一维索引
2. 做越界保护
3. 返回 `BasicMapData[index].Empty`

对应代码：`WorldState.cs:176-185`

因此，大部分放置预检都只依赖基础层，不依赖详细层。

### 6.3 单格综合查询

`GetCell` 会同时返回两部分信息：

- 返回值：`BasicCellData`
- `out hasDetail` + `out detailedData`：详细层是否存在以及内容

对应代码：`WorldState.cs:304-319`

调用方通常会按以下方式使用：

1. 先看 `BasicCellData` 判断土地性质
2. 再看 `hasDetail` / `detailedData` 判断是否有农田、作物或其他实体

## 7. 放置流程

### 7.1 多格设施放置

`PlaceEntity(Vector3Int cellPos, int itemID)` 用于放置普通设施：

1. 通过 `ItemRegistry` 取物品定义
2. 从物品特性里找到 `Feature_Placement`
3. 计算中心点并实例化预制体
4. 遍历 `Length * Height` 覆盖范围
5. 把覆盖到的 `BasicMapData[index].Empty = false`
6. 为格子写入 `DetailedMapData`
7. 调用 `RegisterEntity` 注册运行时实体

对应代码：`WorldState.cs:186-223`

这个流程说明：

- 多格物体的占用判定记录在基础层
- 多格物体与格子的关联记录在详细层
- 运行时行为则进入 `Entitys`

### 7.2 单格瓦片实体放置

`PlaceTile` 更适合“带运行时逻辑的单格瓦片实体”，比如农田和作物：

1. 根据 `tileLayer` 选择 Tilemap
2. 调用 `SetTile`
3. 确保该格有 `DetailedMapData`
4. 把 `nextEntityId` 注册给对应实体

对应代码：`WorldState.cs:259-283`

农田和作物都走了这条通道：

- 锄地生成 `Farmland_Entity`：`WorldState.cs:486-489`
- 种植生成 `Crops_Entity`：`Farmland_Entity.cs:78-103`

## 8. 农田与作物如何落在地图上

### 8.1 农田

锄头交互发生在 `ItemInteract` 中：

1. 读取目标格 `BasicCellData`
2. 当 `Type == Soil` 且 `CheckEmpty == true` 时，允许锄地
3. 创建 `Farmland_Entity`
4. 在第 `1` 层写入 `farmlandTile`
5. 注册成一个地图实体

对应代码：`WorldState.cs:478-490`

农田本体保存的运行时状态主要包括：

- `WaterTimeLeft`
- `FertTimeLeft`
- `FertRank`
- `CropInstanceId`

对应代码：`Farmland_Entity.cs:18-30`

### 8.2 作物

播种时流程为：

1. 通过格子上的 `DetailedCellData` 找到农田实体
2. 判断 `Farmland_Entity.CanPlant`
3. 调用 `Farmland_Entity.Plant`
4. 在第 `2` 层放置种子/作物图块
5. 注册 `Crops_Entity`
6. 把作物实体 ID 反写回 `CropInstanceId`

对应代码：

- 查农田：`Feature_Seed.cs:15-33`
- 播种主体：`Farmland_Entity.cs:78-103`

作物自身不直接保存格子字典，而是依赖：

- `EntityId`
- 关联的 `Farmland_Entity`
- `farmland_Entity.GridPos`

生长阶段变化时，会通过：

```csharp
WorldState.Instance.SwitchTile(farmland_Entity.GridPos, cropTiles[tileIndex], 2);
```

更新第 `2` 层图块。对应代码：`Crops_Entity.cs:58-84`

## 9. 交互检索方式

`InteractAt` 与 `DetectInteract` 都走同一种查找逻辑：

1. 用格子坐标查 `DetailedMapData`
2. 取出该格上的 `EntityID` 列表
3. 逐个通过 `GetEntity` 找到运行时实体
4. 如果实体实现了 `IInteractable`，则执行对应交互

对应代码：

- `InteractAt`：`WorldState.cs:435-452`
- `DetectInteract`：`WorldState.cs:453-470`

这说明当前地图交互的入口不是 Tilemap，而是：

`格子坐标 -> DetailedMapData -> EntityID -> Entitys -> 运行时实体行为`

## 10. 当前方案的整体特点

### 优点

- 查询简单直接，适合当前中小地图
- 基础层与扩展层分离，空地成本低
- Tilemap 只做显示，逻辑不绑死在 Unity 瓦片资源上
- 实体行为通过 `IEntityRuntime` 解耦，后续可扩展更多设施/作物

### 局限

- 还不是完整存档方案，只是运行时内存方案
- 基础层、详细层、Tilemap 显示层存在同步风险
- 多格实体删除、格子类型回写、详细状态落盘等部分还未彻底闭环

如果你只记一件事，可以记成一句话：

当前地图本质上是“基础数组 + 稀疏字典 + 实体注册表 + 三层 Tilemap 显示”的组合方案。
