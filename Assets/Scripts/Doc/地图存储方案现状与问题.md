# 地图存储方案现状与问题

## 1. 当前实现已经成立的部分

从 `WorldState.cs` 及直接调用链来看，当前方案已经具备以下明确特征：

### 1.1 逻辑地图与显示地图已经分离

- `BasicMapData` 保存基础占用与地块类型
- `DetailedMapData` 保存稀疏的附加状态
- `MainTile / OverlapTile / UperTile` 仅负责显示

这意味着后续即使更换瓦片表现，逻辑层仍可独立演进。

### 1.2 实体采用“ID + 运行时对象”管理

当前不是把完整对象直接塞到格子里，而是：

1. 格子保存 `EntityID`
2. `WorldState.Entitys` 保存 `IEntityRuntime`
3. 用 `GetEntity` 做二次跳转

好处是：

- 一个格子可以容纳多个实体 ID
- 地图索引层比较轻
- 运行时行为不需要耦合到地图数组结构

对应代码：

- `DetailedCellData.EntityID`：`WorldState.cs:35`
- `Entitys`：`WorldState.cs:86`
- `RegisterEntity`：`WorldState.cs:143-161`

### 1.3 农田和作物已经形成两层实体结构

当前单格耕作相关逻辑是分层的：

- 农田实体负责水分、肥力、种植位
- 作物实体负责成长、产物、阶段图

二者共同作用于同一个格子，但各自承担不同职责：

- `Farmland_Entity` 是“容器层”
- `Crops_Entity` 是“内容层”

这对未来扩展很有帮助，比如：

- 同一地块允许不同作物逻辑
- 农田和作物分别存档
- 收获时只销毁作物，不销毁农田

## 2. 当前实现中的关键问题

下面这些问题不是“风格建议”，而是会直接影响地图状态正确性的点。

### 2.1 `BasicCellData` 是结构体，但锄地后没有写回数组

`ItemInteract` 中的逻辑是：

```csharp
BasicCellData cell = GetCell(targetGridPos, out bool hasDetail, out DetailedCellData detailedData);
cell.Type = GridType.Farmland;
```

对应代码：`WorldState.cs:480-489`

问题在于 `BasicCellData` 是 `struct`，`GetCell` 返回的是值拷贝，不是引用。

结果：

- 这里修改的是局部变量 `cell`
- `BasicMapData[index]` 里的真实数据没有被改掉
- 视觉上锄地成功，但逻辑层的 `Type` 仍可能还是 `Soil`

这会导致：

- 后续依赖 `Type` 的判断不可靠
- 地块状态与 Tilemap 表现不一致

### 2.2 多格实体删除时，只清理了 pivot 格

`DestroyEntity` 的逻辑只使用：

```csharp
Vector3Int pivotPos = entityRuntime.PivotPos;
```

然后只在 `DetailedMapData[pivotPos]` 中删除该实体 ID。对应代码：`WorldState.cs:419-433`

问题是多格实体在放置时会遍历整个占用范围，并对多个格子写入占用信息：`WorldState.cs:202-220`

但删除时没有反向遍历完整覆盖区域。

结果可能是：

- 其他占用格的 `DetailedMapData` 没清
- 其他占用格的 `BasicMapData[index].Empty` 没恢复
- 物体删掉后，地图仍残留“被占用”状态

### 2.3 `PlaceEntity(string itemID)` 没有补写 `DetailedMapData`

`PlaceEntity(Vector3Int cellPos, string itemID)` 与 `int itemID` 版本并不一致。

`int` 版本里至少有：

```csharp
ApplyDetailedMapData(pos, nextEntityId);
BasicMapData[index].Empty = false;
```

对应代码：`WorldState.cs:215-220`

但 `string` 版本只有：

```csharp
BasicMapData[index].Empty = false;
```

对应代码：`WorldState.cs:241-256`

这意味着通过字符串 ID 放置出来的实体：

- 可能没有进入 `DetailedMapData`
- 之后不能被 `InteractAt` / `DetectInteract` 检出

### 2.4 `PlaceEntity(int itemID)` 里已经明确留了 TODO

当前源码里直接有这一句：

```csharp
Debug.LogError($"你这家伙，忘了写这块的TODO了！");
```

对应代码：`WorldState.cs:215`

这说明作者自己也已经确认，当前多格设施写入详细层的逻辑还没有彻底完成。

虽然现在 `ApplyDetailedMapData(pos, nextEntityId)` 已经会给每个格子创建一条带实体 ID 的记录，但从注释语义上看，原计划可能还想处理更完整的“多格实体反查”信息。

### 2.5 `DestroyEntity` 不会恢复 `BasicMapData[index].Empty`

当前销毁实体只做了：

- 从 `Entitys` 删除
- 从 `DetailedMapData[pivotPos].EntityID` 删除

对应代码：`WorldState.cs:419-433`

但没有做：

- 占用格 `Empty = true`
- 地块类型恢复
- Tilemap 清理

后果是：

- 运行时实体消失了
- 但逻辑占用可能还在
- 新物体可能无法再放置

### 2.6 `DetailedCellData` 中的水肥字段没有成为唯一真源

`DetailedCellData` 定义了：

- `WaterTimeLast`
- `FertTimeLast`
- `FertRank`

对应代码：`WorldState.cs:35-43`

但实际农田运行时状态在 `Farmland_Entity` 中又有一套：

- `WaterTimeLeft`
- `FertTimeLeft`
- `FertRank`

对应代码：`Farmland_Entity.cs:20-23`

当前交互里：

- 浇水时实际调用的是 `farmland.Water()`：`WorldState.cs:501-505`
- 施肥时实际调用的是 `farmland.ApplyFertilizer(1)`：`Feature_Fertilizer.cs:9-22`

这说明：

- 详细层定义了地块状态字段
- 但当前真正生效的是实体内状态
- 两边并没有统一同步机制

如果将来做存档，必须先确定到底谁是权威数据源。

### 2.7 `GetCell` 缺少越界保护

`GetCell` 会直接：

```csharp
int index = Index(target.x, target.y);
return BasicMapData[index];
```

对应代码：`WorldState.cs:304-318`

这里没有像 `CheckEmpty` 一样做边界检查。

如果调用方传了非法坐标，就会直接数组越界。

### 2.8 Tilemap 初始状态不会自动回灌到逻辑地图

`InitMap` 只初始化数组默认值：`WorldState.cs:133-141`

它不会：

- 扫描 `MainTile` 里的已有瓦片
- 根据场景资源自动生成 `BasicMapData`
- 根据现有 Tilemap 反建 `DetailedMapData`

这表示当前系统默认前提是：

- 逻辑地图由代码创建
- 场景 Tilemap 只是空壳或静态参考

如果以后要让美术直接在 Tilemap 上预铺地形，这部分还不够。

### 2.9 种子和肥料默认取 `EntityID[0]`，有顺序假设

`Feature_Seed` 和 `Feature_Fertilizer` 都直接取：

```csharp
int entityID = detailedData.EntityID[0];
```

对应代码：

- `Feature_Seed.cs:22`
- `Feature_Fertilizer.cs:12`

但 `DetailedCellData.EntityID` 是一个列表，不保证第一个一定是农田。

一旦同格里同时存在多个实体，例如：

- 农田
- 作物
- 未来的地表特效或装置

那么固定取第一个就会变得脆弱。

## 3. 当前真实的地图存储分层

如果按“谁在负责什么”来重新表述，当前实际上是四层结构：

### 3.1 基础地块层

载体：`BasicMapData`

职责：

- 土地类型
- 占用判定

### 3.2 稀疏附加层

载体：`DetailedMapData`

职责：

- 某格有哪些实体 ID
- 预留的水肥扩展字段

### 3.3 运行时实体层

载体：`Entitys`

职责：

- 实体行为
- 作物成长
- 农田含水
- 交互响应

### 3.4 显示层

载体：三层 Tilemap

职责：

- 地表显示
- 农田覆盖
- 作物阶段表现

所以项目现在不是“只有一套地图数据”，而是“逻辑、扩展、实体、显示”四层并存。

## 4. 建议落地方向

如果你后面要把这套系统发展成稳定的地图存档方案，建议优先统一下面三件事。

### 4.1 先确定唯一真源

推荐明确以下分工：

- `BasicMapData` 作为所有格子的基础真源
- `DetailedMapData` 作为稀疏格子状态真源
- `Entitys` 只做运行时对象，不直接承担长期存档真源

这样存档时就可以：

1. 序列化基础层数组
2. 序列化详细层字典
3. 根据详细层里的实体数据重建运行时对象

### 4.2 给多格实体补完整占用信息

建议每个可放置实体至少保存：

- `PivotPos`
- `Size`
- 或完整 `OccupiedCells`

否则删除、存档、加载时都很难准确恢复。

### 4.3 把“状态修改”封装成统一接口

例如增加：

- `SetCellType`
- `SetCellEmpty`
- `AddEntityToCell`
- `RemoveEntityFromCell`
- `SetFarmlandMoisture`

避免外部代码直接零散改：

- `BasicMapData[index]`
- `DetailedMapData[targetGridPos]`
- Tilemap

这样才能保证逻辑层和显示层同步。

## 5. 如果要做存档，建议的序列化最小集合

基于当前代码，最小可用的地图存档数据建议包含：

### 5.1 基础格子数据

- 地图宽高
- 每个格子的 `GridType`
- 每个格子的 `Empty`

### 5.2 详细格子数据

- 格子坐标
- `EntityID` 列表
- 水分剩余时间
- 肥料剩余时间
- 肥料等级

### 5.3 实体数据

至少需要：

- `EntityId`
- 实体类型
- `PivotPos`
- 多格实体占用范围
- 该实体的专属运行时状态

对于农田和作物，还需要额外保存：

- 农田的水分、肥力、当前作物 ID
- 作物的成长时间、是否可收获、产物信息

## 6. 一句话总结

当前地图方案已经有了比较清晰的雏形：

- 用数组管全图基础状态
- 用字典管稀疏格子扩展状态
- 用实体表管运行时行为
- 用 Tilemap 管显示

但它目前仍处在“运行时可用、存档闭环未完成”的阶段，尤其需要优先补齐结构体回写、多格实体删除、详细层一致性和唯一真源这几项关键逻辑。
